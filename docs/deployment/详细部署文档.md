# SmartClass Agent

SmartClass Agent 是一个面向教师的多模态互动式教学智能体。它把备课、教学设计、知识库检索、附件分析、产物生成与产物迭代修改串成一条完整工作流，帮助教师从一个教学主题出发，逐步沉淀为可预览、可下载、可继续修改的教学材料。

项目采用前后端分离架构，主仓库通过 Git submodule 组合后端、前端与落地页子仓库，并提供本地源码部署与 Docker Compose 一体化部署两种方式。

## 亮点

- **面向真实备课流程**：支持教学要素抽取、信息缺口追问、教学设计确认、产物选择与审批恢复。
- **多模态上下文输入**：支持知识库文件、会话附件、语音转写与视频附件分析。
- **RAG 与长期记忆**：基于 PostgreSQL/pgvector 风格检索增强，结合 LangGraph store 保存用户画像与可复用教学经验。
- **可交付产物闭环**：可生成和修改 PPT 课件、DOCX 教案、HTML 互动内容，并保留 revision 版本关系。
- **受控 Agent Workspace**：产物子 Agent 在受控 workspace 中执行 skill 和代码工具，限制路径访问、安装命令、执行超时与输出长度。
- **对象存储抽象**：通过 `StorageService` 统一本地存储与 MinIO，对知识库、附件、产物使用稳定 `storage_key`。
- **登录态与资源隔离**：JWT Bearer 认证，计划、会话、文件、产物按 `user_id` 与业务上下文过滤。
- **工程可观测性**：支持结构化日志、可选 JSONL trace、OpenTelemetry OTLP、Prometheus `/metrics` 与 Grafana dashboard。
- **Agent 评估系统**：`backend/tests/evals/` 已包含 intent、memory、extraction 等评估用例、评估器、runner 与 JSON report。

## 快速启动（Docker Compose）

推荐先用 Docker Compose 跑通完整链路。Compose 栈包含前端、后端、PostgreSQL/pgvector、MinIO、OnlyOffice、OpenTelemetry Collector、Prometheus 和 Grafana。

1. 准备环境文件：

```powershell
Copy-Item .env.docker.example .env.docker
```

2. 编辑 `.env.docker`，至少填写：

- `JWT_SECRET_KEY`
- `POSTGRES_PASSWORD`
- `MINIO_ROOT_PASSWORD`
- `MINIO_SECRET_KEY`
- `GRAFANA_ADMIN_PASSWORD`
- `MODEL` / `API_KEY` / `BASE_URL`
- `EMBEDDINGS_MODEL` / `EMBEDDINGS_API_KEY` / `EMBEDDINGS_BASE_URL`
- 如需语音或视频能力，再配置 `STT_*`、`VIDEO_VISION_*`

3. 构建并启动：

```powershell
docker compose --env-file .env.docker up -d --build
```

4. 访问默认地址：

| 服务 | 地址 |
| --- | --- |
| SmartClass 前端 | http://localhost:8080 |
| 后端健康检查 | http://localhost:8000/health |
| MinIO Console | http://localhost:9001 |
| OnlyOffice | http://localhost:8082 |
| Prometheus | http://localhost:9090 |
| Grafana | http://localhost:3000 |

5. 可选：运行基础检查脚本。

```powershell
.\docker\verify.ps1
```

脚本会检查 Compose 配置、前后端健康状态、OnlyOffice 脚本、PostgreSQL vector 扩展、后端运行时工具、Prometheus target 与 Grafana health。更多 Docker 细节见 [docs/deployment/docker.md](docs/deployment/docker.md)。

停止服务：

```powershell
docker compose --env-file .env.docker down
```

如需清空数据库、MinIO、Grafana 等持久化数据：

```powershell
docker compose --env-file .env.docker down -v
```

## 源码部署

源码部署适合开发调试。你需要自行准备 PostgreSQL、模型服务、可选 MinIO、可选 OnlyOffice，以及后端文档/音视频处理依赖。

### 1. 环境要求

- Python 3.12（或与 `backend/requirements.txt` 兼容的 Python 3.11+ 环境）
- Node.js 20（前端构建与后端产物 skill 都会用到）
- PostgreSQL，建议启用 `vector` 扩展
- 可选：MinIO，用于对象存储模式
- 可选：OnlyOffice Document Server，用于 Office 文档预览与回写
- 可选：ffmpeg、LibreOffice、Poppler，用于视频/文档处理能力

### 2. 准备根目录环境变量

后端会读取主仓库根目录的 `.env`：

```powershell
Copy-Item .env.local.example .env
```

按你的本地环境填写：

- 数据库：`DATABASE_URL` 或 `DB_HOST` / `DB_PORT` / `DB_USER` / `DB_PASSWORD` / `DB_NAME`
- 模型：`MODEL`、`API_KEY`、`BASE_URL`
- Embedding：`EMBEDDINGS_MODEL`、`EMBEDDINGS_API_KEY`、`EMBEDDINGS_BASE_URL`
- JWT：`JWT_SECRET_KEY`
- 存储：`STORAGE_BACKEND=local` 或 `STORAGE_BACKEND=minio`
- OnlyOffice 相关 URL：根据部署方式设置 `PUBLIC_API_BASE_URL`

本地只验证基础聊天和知识库能力时，可以先使用 `STORAGE_BACKEND=local`。如果要验证接近生产的文件链路，建议使用 MinIO。

### 3. 启动后端

在主仓库根目录执行：

```powershell
cd backend
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install -r requirements.txt
npm install
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
```

后端启动时会初始化数据库表、LangGraph checkpoint/store、RAG runtime、skill registry、文件入库 runtime 与 Agent runtime。

健康检查：

```powershell
Invoke-WebRequest http://localhost:8000/health
```

### 4. 启动前端

另开终端，回到主仓库根目录执行：

```powershell
cd frontend
npm install
npm run dev
```

Vite 开发服务器默认会把 `/api` 代理到 `http://127.0.0.1:8000`。OnlyOffice 相关路径在当前 `frontend/vite.config.js` 中使用了固定代理地址；如果你要在源码模式下本地验证 OnlyOffice，请按你的 OnlyOffice 地址调整 `/web-apps`、`/sdkjs`、`/coauthoring`、`/cache`、`/savefile` 代理，或者直接使用 Docker Compose 默认栈。

### 5. 常用源码验证

以下命令分别从主仓库根目录执行：

```powershell
# 后端测试
cd backend
.\.venv\Scripts\python.exe -m pytest
cd ..

# 前端构建
cd frontend
npm run build
cd ..

# 评估系统命令帮助
cd backend
.\.venv\Scripts\python.exe -m tests.evals.cli --help
```

评估用例会实际运行 Agent，通常需要数据库、模型 API 和相关环境变量完整可用。

## 主要配置

| 配置 | 说明 |
| --- | --- |
| `DATABASE_URL` / `DB_*` | PostgreSQL 连接。Docker 默认指向 `postgres:5432` |
| `JWT_SECRET_KEY` | JWT 签名密钥，部署时必须替换 |
| `MODEL`, `API_KEY`, `BASE_URL` | 主聊天模型，OpenAI 兼容接口 |
| `STRUCTED_MDOEL`, `STRUCTED_API_KEY`, `STRUCTED_BASE_URL` | 结构化输出模型。变量名沿用当前代码中的拼写 |
| `SMALL_MDOEL`, `SMALL_API_KEY`, `SMALL_BASE_URL` | 小模型/快速模型，用于 fallback 与后续建议等 |
| `MEMORY_MODEL`, `MEMORY_API_KEY`, `MEMORY_BASE_URL` | 长期记忆反思与选择模型，可回退到结构化/小模型/主模型 |
| `EMBEDDINGS_MODEL`, `EMBEDDINGS_API_KEY`, `EMBEDDINGS_BASE_URL` | RAG 向量化模型 |
| `STT_*`, `VIDEO_VISION_*` | 语音转写与视频视觉分析 |
| `STORAGE_BACKEND` | `local` 或 `minio` |
| `STORAGE_DOWNLOAD_MODE` | Docker 推荐 `proxy`，由后端统一鉴权下载 |
| `PUBLIC_API_BASE_URL` | 后端生成下载、预览、OnlyOffice callback URL 时使用的浏览器可访问源 |
| `WORKSPACE_EXECUTION_BACKEND` | `local` 或 `daytona` |
| `OTEL_ENABLED`, `PROMETHEUS_ENABLED` | 控制 OpenTelemetry 与 Prometheus 导出 |

更多变量见 [.env.local.example](.env.local.example) 和 [.env.docker.example](.env.docker.example)。

## 可观测性

后端可观测性入口位于 `backend/app/core/observability.py` 和 `backend/app/core/observability_bootstrap.py`：

- `RunContext` 统一携带 `run_id`、`thread_id`、`plan_id`、`user_id`、`agent_name`
- `ObservationSink` 支持结构化 logging、JSONL trace、OpenTelemetry span、Prometheus metrics
- `trace_span()`、`record_metric()`、`log_observation()` 与 `observe_llm_call()` 用于记录节点耗时、工具调用、RAG、存储、产物、失败类型和 token usage
- Prometheus 默认指标路径为 `/metrics`
- Docker 栈内 Prometheus 会抓取 `backend:8000/metrics`
- Grafana 已配置 Prometheus datasource 与 starter dashboard

观测字段会经过脱敏与截断。不要把完整 prompt、completion、附件正文、RAG chunk、长期记忆正文、JWT、Authorization header、预签名 URL 签名、对象 key 或宿主机路径写入观测事件。

更多说明见 [docs/observability/README.md](docs/observability/README.md)。

## 评估系统

Agent 评估系统位于 `backend/tests/evals/`，用于检查行为是否在迭代中保持稳定：

```text
backend/tests/evals/
├─ cases/        # YAML 评估用例
├─ evaluators/   # intent / memory / extraction 评估器
├─ runners/      # EvalRunner
├─ rubrics/      # rubric 配置
├─ results/      # JSON report 输出目录
└─ cli.py
```

常用命令：

```powershell
cd backend
python -m tests.evals.cli list-categories
python -m tests.evals.cli validate tests/evals/cases/intent/basic_chat.yaml
python -m tests.evals.cli run --category intent_recognition
python -m tests.evals.cli run --case-id intent_basic_chat_001 --verbose
```

当前已覆盖或初步覆盖意图识别、记忆检索/写入/更新、教学要素抽取等维度。产物生成/修改、安全性、性能与 CI 质量门禁仍适合继续扩展。

## API 概览

后端统一挂载在 `/api` 下：

- `POST /api/auth/register`、`POST /api/auth/login`、`GET /api/auth/me`
- `POST /api/chat/stream`
- `/api/plan`、`/api/session`
- `/api/file/knowledgeFile`、`/api/file/attachment/*`、`/api/file/artifact`
- `/api/memory`

`/api/chat/stream` 使用 SSE 推送运行事件，当前事件包括 `metadata`、`token`、`progress`、`artifact`、`artifact_trace`、`approval`、`suggestions`、`error`、`done`。

## 生产部署建议

- 只公开前端或统一 ingress 域名；PostgreSQL、MinIO、Prometheus、Grafana、OTel Collector 不应直接暴露公网。
- 所有默认密码和密钥必须替换，`.env`、`.env.docker` 不要提交到 Git。
- 生产建议使用 HTTPS，并正确设置 `PUBLIC_API_BASE_URL`。
- MinIO 下载建议保持 `STORAGE_DOWNLOAD_MODE=proxy`，除非你有明确的公网 S3/预签名 URL 策略。
- 为 PostgreSQL、MinIO 和必要的 backend storage volume 建立备份。
- 高流量环境降低 `OTEL_TRACES_SAMPLER_ARG`，避免全量 trace 带来成本和存储压力。
- Agent 长任务、文件入库和产物生成在进一步规模化时，建议拆分独立 worker 与任务队列。
