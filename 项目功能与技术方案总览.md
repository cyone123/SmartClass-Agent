# SmartClass 项目功能与技术方案总览

## 1. 项目定位

SmartClass 是一个面向教师场景的多模态教学智能体系统，目标不是只做“问答聊天”，而是把教师备课过程中的多个关键环节串起来，形成从资料管理、教学要素提取、教学设计规划，到课件/教案/互动内容生成，再到产物修改与预览的完整闭环。

从当前代码实现来看，这个项目已经具备较完整的教学智能体骨架，尤其强调以下几个方向：

- 以教师备课与教学设计为核心，而不是通用聊天机器人
- 支持知识库文件、会话附件、语音输入、视频附件等多种输入方式
- 用 LangGraph 编排真实工作流，而不是把所有逻辑塞进一个 prompt
- 将生成结果抽象为 artifact，并围绕 artifact 建立生成、展示、编辑、修订的闭环
- 前端不是简单文本对话，而是具备流式消息、进度卡片、审批卡片、产物追踪和文档预览能力

一句话概括：这是一个“教学对话 + RAG + 多模态附件分析 + 教学设计 + 可编辑产物生成与修订”的教师智能工作台。

## 2. 当前系统全貌

### 2.1 面向用户的核心能力

当前项目已经实现或基本打通了以下能力：

- 教学计划与教学会话管理
- 知识库文件上传、异步解析、向量入库与检索
- 会话附件上传与按轮次注入上下文
- 语音输入转写并回填到对话输入框
- 视频附件的音频转写、关键帧抽取与画面摘要
- 基于 LangGraph 的教学意图识别、教学元数据提取、追问补全、RAG 检索、教学设计输出
- 教学设计确认后按需生成 PPT、DOCX 教案、HTML 互动内容
- 对已生成产物进行差量修改，而不是只能重新生成
- 产物在线展示、HTML 预览、OnlyOffice 文档预览与回写
- 前端实时展示 token、流程进度、审批节点、产物状态和产物生成轨迹

### 2.2 当前阶段判断

从仓库实现状态看，这个项目已经不是“概念验证 Demo”，也还不是“完全产品化平台”。它更像是一个已经打通主链路的智能教学 Agent 原型系统，重点价值在于：

- 核心工作流已经存在，且前后端协议已基本成形
- 产物生成和产物修订已经进入真实工程实现，不是停留在文本回复层
- 多模态输入能力已经接入到统一对话流程中
- 系统下一阶段最重要的方向，是继续完善产物修订、协议统一和稳定性，而不是从零搭功能

## 3. 总体架构

项目采用典型的前后端分离结构：

- 前端：Vue 3 + Vite + Pinia + Element Plus
- 后端：FastAPI + LangChain + LangGraph + SQLAlchemy + PostgreSQL
- 检索：PGVector 风格向量检索
- 模型接口：OpenAI 兼容接口
- 文档编辑/预览：OnlyOffice

整体可以理解为四层：

### 3.1 交互层

由前端负责承接教师操作，包括：

- 教学计划与会话切换
- 对话输入
- 附件上传
- 语音录音
- 进度与审批展示
- 产物查看与预览

### 3.2 编排层

由 LangGraph 负责工作流编排，核心节点包括：

- 意图识别
- 教学元数据提取
- 追问补全
- 元数据确认
- RAG 检索
- 教学设计生成
- 教学设计确认
- 产物生成
- 产物修订路由
- 产物结果汇总

### 3.3 能力层

由多个专门运行时和工具链提供能力：

- RAG 运行时
- 附件分析 Agent
- Skill Registry
- Workspace 执行环境
- 语音转写运行时
- 视频分析运行时
- Artifact 生成/修订 Agent

### 3.4 数据与存储层

由数据库和文件存储共同承载：

- PostgreSQL 存储计划、会话、文件、artifact 元数据
- LangGraph checkpoint 存储会话状态和中断恢复信息
- 本地存储目录保存知识库文件、附件、artifact、agent workspace
- PGVector 表保存向量化后的知识切片

## 4. 技术栈与作用分工

| 技术 | 作用 |
| --- | --- |
| FastAPI | 提供聊天、文件、计划、会话等 API |
| LangChain | 构建模型调用、Agent、工具调用能力 |

| PostgreSQL | 业务数据与会话状态持久化 |

## 5. 后端功能实现与技术方案

### 5.1 会话与教学计划管理

后端提供教学计划和教学会话的基本 CRUD 接口：

- `plan` 负责教学计划管理
- `session` 负责会话管理，并用 `thread_id` 作为 LangGraph 会话标识

这里的设计意义在于：

- `plan_id` 作为知识库、附件、artifact 的上层归属
- `thread_id` 作为多轮对话状态、审批中断、artifact 历史的主索引

这使得“教学计划”和“具体会话”形成了清晰的双层结构，便于把知识库、公用资料和具体一次备课过程区分开。

### 5.2 智能对话主链路

核心入口是 `POST /api/chat/stream`，采用 SSE 流式返回。

这条链路的典型执行过程是：

1. 前端提交消息、`thread_id`、附件 ID 或审批信息
2. 后端根据 `thread_id` 找到会话与 `plan_id`
3. 若本轮携带附件，先完成附件分析，得到可注入上下文
4. 将用户消息、附件摘要和已有会话状态送入 LangGraph
5. LangGraph 根据当前状态和意图进入不同分支
6. 运行过程中持续通过 SSE 向前端推送 token、progress、artifact、approval 等事件

当前实际支持的 SSE 事件比最初目标契约更丰富，包含：

- `metadata`
- `progress`
- `token`
- `artifact`
- `artifact_trace`
- `approval`
- `suggestions`
- `error`
- `done`

这说明项目前后端之间已经不是“只推文本”的协议，而是一个更接近工作流 UI 的事件协议。

### 5.3 LangGraph 教学工作流

项目最核心的技术实现之一，是用 LangGraph 将教学流程拆成显式节点，而不是用单次大模型调用硬做。

当前图中包含三类主路径：

#### 1. 普通聊天路径

- 意图识别为 `normal_chat`
- 直接调用主模型进行普通对话

#### 2. 教学设计路径

- 意图识别为 `teaching_plan`
- 结构化提取教学元数据
- 元数据不完整时自动追问
- 进入中断节点，等待用户补充
- 元数据完整后进入确认节点
- 用户确认后执行 RAG 检索
- 基于元数据和检索结果生成教学设计方案
- 再次进入确认节点
- 用户确认后按需生成不同类型 artifact

#### 3. 产物修订路径

- 意图识别为 `artifact_revision`
- 从当前线程的现有 artifact 中推断修订目标
- 如果修订对象不明确，进入澄清型审批中断
- 如果目标明确，则进入对应 artifact 的修订节点
- 最后在 fan-in 节点统一汇总结果

这种实现方式的价值在于：

- 工作流状态可恢复
- 节点职责清晰
- 容易插入审批、中断、补充输入等人机协作环节
- 便于后续继续扩展更多产物类型和更多多模态处理节点

### 5.4 结构化输出与稳态优化

项目没有把“意图识别”和“教学元数据提取”写成自由文本解析，而是采用了结构化输出：

- `ConversationRoute` 结构化识别意图
- `TeachingMetadata` 结构化提取学科、年级、主题、课时、重点、难点、目标等字段

实现上使用了：

- `json_schema` 方式的结构化输出
- fast 模型优先、fallback 模型兜底
- schema warmup 预热
- 可选 prompt cache
- structured output 延迟日志记录

这部分设计体现了很强的工程意识：项目不仅在“能不能抽取”，还在关注结构化调用的稳定性、首调用开销和故障兜底。

### 5.5 RAG 知识库能力

知识库链路已经实现为独立流程：

1. 文件上传到 `knowledge_files`
2. 写入本地存储
3. 异步加入入库队列
4. `FileIngestionRuntime` 后台消费
5. 文档切分、附加元数据、向量写入 PGVector
6. 在教学设计阶段按 `plan_id` 过滤检索

当前代码层面支持的知识库文件类型包括：

- PDF
- DOCX
- TXT
- MD / Markdown
- CSV
- JSON

其中：

- PDF 使用 `PyPDFLoader`
- DOCX 直接解析 `word/document.xml`
- 文本类文件做编码兜底读取
- 统一使用 `RecursiveCharacterTextSplitter` 切分

这条链路把知识库文件和会话附件明确区分开，是项目的重要架构约束之一。

### 5.6 附件分析链路

会话附件不进入知识库，而是用于“当前对话上下文增强”。

当前附件链路分为两类：

- 文档类附件：交给附件分析 Agent，通过 Skill 分析内容后生成摘要
- 视频类附件：走视频分析运行时，生成音频转写和画面摘要

最终统一输出为可注入上下文文本，再作为 system/human 消息的一部分送入图流程。

这体现了一个很好的抽象：多模态输入最终都被折叠成“可注入上下文”，从而与图编排和 Prompt 层兼容。

### 5.7 语音输入与视频理解

项目已经实现两条多模态链路：

#### 语音输入

- 前端浏览器录音
- 本地编码为 WAV
- 调用 `/file/attachment/voice/transcribe`
- 后端调用 OpenAI 兼容转写接口
- 返回文本并回填到输入框

这里的定位更像“语音转文本输入增强”，而不是直接做音频对话。

#### 视频附件分析

视频分析运行时会执行：

- FFmpeg 抽取音频
- 调用语音转写模型
- FFmpeg 抽取关键帧
- 调用视觉模型对关键帧做画面转述
- 合并为视频附件摘要

最终摘要既包含“音频转写”，也包含“关键画面摘要”，适合教学场景下对课堂演示视频、实验录像、讲解视频等资料做快速理解。

### 5.8 Skill 体系与 Agent 工具治理

项目实现了一个比较完整的 Skill 机制，而不是只靠 prompt 约束：

- `SkillRegistry` 读取 `skills/*/SKILL.md`
- 解析 frontmatter 中的名称、描述、兼容性、允许工具等元数据
- 提供 `load_skill`、`read_skill_resource`、`run_skill_script` 三类工具

当前已接入可用的 skill 包括：

- `pdf`
- `docx`
- `ppt-generator`
- `html-interactive`

同时，项目引入了明确的执行治理策略：

- 只有加载了 skill，且 skill 显式允许，才能使用 workspace 代码工具
- 禁止通过 `shell` 执行 Python/Node/npm/pip 等安装或代码运行命令
- 优先用 skill 脚本和 workspace 工具，而不是任由模型直接做不受控操作

这是一种很有价值的工程设计：既给了 Agent 足够执行能力，又通过 skill 元数据和中间件把能力边界收紧。

### 5.9 Workspace 执行环境

artifact 生成与修订依赖一个隔离的 workspace 执行环境。

每个线程/运行都会映射出：

- `workspace_root`
- `run_root`
- `output_root`

系统提供五类工作区工具：

- 列出文件
- 读取文件
- 写入文件
- 精准替换文本
- 运行 Python/Node 代码

同时带有明确安全控制：

- 禁止路径穿越
- 禁止依赖安装命令
- 限制输出长度
- 对进度做结构化上报

这使得 artifact Agent 不只是“写回复”，而是真的可以在受控环境中读写文件并生成实际产物。

### 5.10 Artifact 生成、存储与修订

artifact 是项目最有辨识度的实现之一。

当前支持三类 artifact：

- `ppt`
- `docx`
- `html-game`

每次生成或修订都会在 `artifact_files` 中留下记录，记录包含：

- 类型
- 标题
- 状态
- 线程与计划归属
- 存储路径
- 父产物 ID
- 根产物 ID
- 修订版本号
- 是否当前版本

这意味着系统已经具备“版本链”而不是简单文件覆盖。

#### 生成流程

- 图流程进入生成节点
- 创建运行中 artifact 记录
- 启动对应的 artifact Agent
- Agent 根据 skill 和 workspace 生成真实文件
- 系统自动发现输出文件
- 将产物拷贝到正式存储路径
- 更新 artifact 状态并通过 SSE 推给前端

#### 修订流程

项目并不是简单“重新生成一版”。

它会先：

- 找到当前线程下可修订 artifact
- 根据用户文本推断修改目标
- 把源文件复制进 workspace
- 对 Office 文件自动解包到 `source_unpacked`
- 生成 `source_summary.json`
- 把修订请求写成 `revision_request.json`

然后再让对应 artifact Agent 依据已有源文件、摘要和用户 delta 做定向修订。

这是项目目前最重要的创新实现之一：系统已经开始支持“基于现有产物的差量修改”，而不是每次都从零开始生成。

### 5.11 Artifact Trace 与过程可视化

artifact Agent 在执行过程中，不仅最终回传结果，还会把过程分解成 trace 条目：

- 状态变化
- AI 输出
- 工具调用
- 工具结果

这些 trace 通过 `artifact_trace` SSE 事件发送给前端，用于还原产物生成/修订过程。

这让产物生成不再是黑盒，对教师或开发者都更友好，也更利于调试和后续流程可解释性建设。

## 6. 前端功能实现与交互方案

### 6.1 三栏工作台结构

前端首页采用典型三栏布局：

- 左侧：教学计划与会话管理
- 中间：教学对话区
- 右侧：知识库文件与会话产物区

这种结构很贴合教师实际使用流程：先选计划，再进入某次备课会话，再在右侧查看资料与生成结果。

### 6.2 教学对话区

聊天区已经不是简单 IM 窗口，而是复合型智能工作流界面，支持：

- 教师消息与 AI 消息
- AI Markdown 实时渲染
- 流式 token 追加
- 进度卡片
- 审批卡片
- artifact trace 卡片
- 附件展示
- 建议追问 chips

前端通过手动解析 SSE 流完成对事件的接收和分发，而不是只监听单一 message 事件。

### 6.3 审批与人机协作界面

项目把“确认教学要素”“确认教学设计方案”“确认修订目标”等中断环节显式做成审批卡片，这一点很关键。

它意味着：

- 教师不是被动接收结果，而是可以在关键节点确认或反馈
- 工作流可以中断、等待、恢复
- 前后端围绕审批节点形成了稳定协议

特别是在教学设计确认节点，前端还允许教师勾选要生成的 artifact 类型，实现选择性生成。

### 6.4 知识库与产物面板

右侧资料面板承接了两类内容：

- 当前会话产物
- 当前教学计划的知识库文件

其中知识库文件支持：

- 上传
- 状态轮询
- 删除
- 预览

artifact 支持：

- 实时刷新状态
- Office/PDF 预览
- HTML 互动内容 iframe 预览

这让对话结果不再停留在“文本结论”，而是进入可查看、可编辑、可继续修改的资料区。

### 6.5 OnlyOffice 集成

OnlyOffice 集成是项目落地性很强的一部分。

前端通过 `/file/config/{file_kind}/{file_id}` 获取配置后初始化编辑器，后端再通过 `/file/callback` 接住回写。

回写后：

- 如果是知识库文件，系统会重新置为待入库并自动重新入库
- 如果是 artifact，则更新存储文件并保持 ready 状态

这使系统具备了“AI 生成 + 人工编辑 + 系统回收更新”的闭环能力。

## 7. 项目亮点与创新点

### 7.1 不止聊天，而是完整教学工作流

很多教学智能体项目停留在“输入一句话，返回一段文本建议”。SmartClass 已经把教师工作流程拆成了可执行的步骤，包括补充信息、确认方案、生成产物、修改产物、预览文档等，工程完整度明显更高。

### 7.2 产物差量修改能力

这是当前项目最值得强调的亮点之一。

系统已经不满足于“重新生成一份 PPT/教案”，而是开始围绕已有产物做：

- 版本链管理
- 源文件注入 workspace
- 源内容摘要
- 解包 XML 参考
- 按目标 artifact 定向修订

这类能力比单次生成更接近真实教学生产场景。

### 7.3 Skill + Workspace 的可控 Agent 执行模型

项目不是放任模型直接调用 shell，而是构建了：

- 可发现的 skill
- 渐进式加载
- 工具权限声明
- workspace 安全边界
- 依赖安装禁令

这套机制兼顾了 Agent 执行能力与工程可控性，是很有实用价值的实现方式。

### 7.4 多模态输入被统一抽象为“可注入上下文”

无论是 PDF、DOCX、语音还是视频，最终都被转成适合 graph/prompt 使用的上下文材料。这种抽象使系统在扩展新模态时不必重写整个工作流，是一个很稳的架构方向。

### 7.5 对话过程可视化程度高

系统不仅展示最终回答，还展示：

- 进度步骤
- 审批节点
- 产物状态
- 产物生成轨迹
- 继续追问建议

这使系统从“黑盒生成器”变成“可观察、可协作的智能流程界面”。

### 7.6 AI 产物与文档编辑工具打通

OnlyOffice 的引入，使得产物不是一次性下载文件，而是进入在线编辑和回写流程。这一步极大提升了项目的实用性和教学办公场景适配度。

## 8. 当前真实边界与注意事项

为了避免把未来规划误写成已实现能力，这里明确几个当前边界：

- 这是一个主链路已打通的系统，但仍处于持续完善阶段
- 知识库入库目前已支持 PDF、DOCX 和多种文本格式，但仍以文本型资料为主
- 语音输入当前定位是“转写回填输入框”，不是完整语音会话助手
- 视频理解当前以音频转写与关键帧摘要为主，不是复杂时序视频推理系统
- artifact 类型目前主要是 PPT、DOCX、HTML 互动内容三类
- 前端侧的账户切换、设置面板等有明显 mock 展示性质，不属于核心智能链路

## 9. 适合作为对外介绍的项目总结

SmartClass 的核心价值，在于它不是把大模型简单接成聊天框，而是围绕教师真实备课流程，构建了一个可协作、可追踪、可编辑、可修订的教学智能体系统。它通过 LangGraph 将教学工作流显式化，通过 RAG 与多模态附件增强上下文，通过 Skill + Workspace 机制驱动真实产物生成，并通过 SSE、审批卡片、artifact trace、OnlyOffice 等机制把整个过程完整呈现在前端。

从产品和工程两方面看，这个项目最突出的特点是：

- 面向教师备课全流程，而不是通用问答
- 具备真实 artifact 生成与修订闭环
- 多模态输入与工作流编排结合紧密
- 前后端协议设计已经开始朝“统一流程协议”演进

如果继续沿当前方向推进，SmartClass 很有潜力从“教学智能体原型”进一步演进为真正的教师智能工作平台。
