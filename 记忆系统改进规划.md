我查了 LangChain/LangGraph 与 OpenAI Agents SDK 的最新资料后，建议把 SmartClass 的“记忆系统”从现在的 **thread 级 checkpoint 会话状态**，升级为一套完整的 **Context Engineering + Memory Harness**。

当前项目已有基础：`LangGraph + AsyncPostgresSaver` 已经能按 `thread_id` 保存会话状态，`messages` 是图状态的一部分；但还缺少长期记忆、跨会话偏好、教学资产记忆、记忆压缩、记忆治理与可删除机制。

**参考依据**
- LangChain Context Engineering：可靠 Agent 的关键常常不是模型不够强，而是“正确上下文没有以正确格式传给模型”；上下文分为 runtime context、state、store 三类。  
  https://docs.langchain.com/oss/python/langchain/context-engineering
- LangChain Short-term Memory：短期记忆应基于 thread/checkpointer，长对话需要 trim、delete、summarize。  
  https://docs.langchain.com/oss/python/langchain/short-term-memory
- LangChain Long-term Memory：长期记忆应跨 thread 存储，基于 namespace/key 的 store，适合用户级、组织级、应用级记忆。  
  https://docs.langchain.com/oss/python/langchain/long-term-memory
- LangChain Memory Concept：长期记忆可分为 semantic、episodic、procedural 三类。  
  https://docs.langchain.com/oss/javascript/concepts/memory
- OpenAI Agents SDK Sessions：session memory 自动维护多轮历史，但仍需要限制历史、压缩与自定义合并策略。  
  https://openai.github.io/openai-agents-python/sessions/
- OpenAI Cookbook：长上下文如果不整理，会带来分心、成本、幻觉和调试困难；推荐 trimming 与 compression。  
  https://developers.openai.com/cookbook/examples/agents_sdk/session_memory
- OpenAI Sandbox Agent Memory：长期记忆应采用渐进披露，先注入摘要，必要时再检索更详细记忆，并允许记忆过期、更新或只读。  
  https://openai.github.io/openai-agents-js/guides/sandbox-agents/memory/

## **一、总体目标**
把当前记忆能力升级为四层：

1. **运行时上下文 Runtime Context**  
   当前用户、学校、角色、权限、模型配置、地区课标、功能开关。

2. **会话记忆 Short-term / Thread Memory**  
   当前 `thread_id` 下的消息、审批状态、附件摘要、当前教学目标、待生成 artifact、未完成任务。

3. **长期记忆 Long-term Memory**  
   跨会话保存教师偏好、常用教材、教学风格、历史备课经验、产物修改偏好、学校模板规则。

4. **知识与资产记忆 Knowledge / Artifact Memory**  
   知识库文件、课程资料、已生成课件/教案/互动内容、版本链、修改记录、可复用教学片段。

一句话：不要再让模型“看完整聊天记录猜上下文”，而是让系统主动组装一个干净、分层、有预算、有来源的 `ContextBundle`。

## **二、建议记忆分类**
为 SmartClass 定义 6 类记忆：

| 类型 | 作用 | 示例 |
|---|---|---|
| `teacher_profile` | 教师长期偏好 | 喜欢探究式导入、语言简洁、偏好表格式教案 |
| `teaching_context` | 教学场景记忆 | 常教初中物理、人教版、班级基础偏弱 |
| `plan_memory` | 教学计划级记忆 | 本单元目标、教材章节、已上传资源摘要 |
| `thread_memory` | 当前会话记忆 | 已确认的元数据、审批选择、待修改产物 |
| `artifact_memory` | 产物记忆 | 某 PPT 的风格、版本链、用户修改意见 |
| `org_memory` | 学校/教研组规则 | 学校教案模板、公开课规范、校本评价标准 |

同时对应三种经典记忆：

- **Semantic Memory**：事实和偏好，例如教师偏好、教材版本、班级特点。
- **Episodic Memory**：过去发生过的教学任务，例如“上次为《浮力》生成过实验导入课件，用户要求减少文字”。
- **Procedural Memory**：做事规则，例如“生成本校教案时必须包含教学反思、核心素养、作业设计”。

## **三、数据模型规划**
建议新增以下表或等价模型：

```ts
type MemoryItem = {
  id: string
  scope_type: "user" | "plan" | "thread" | "artifact" | "org" | "agent"
  scope_id: string
  memory_type: "semantic" | "episodic" | "procedural"
  category: string
  content: string
  structured_value?: object
  source_type: "chat" | "approval" | "artifact_revision" | "file" | "manual"
  source_id?: string
  confidence: number
  importance: number
  status: "active" | "archived" | "deleted"
  created_at: string
  updated_at: string
  expires_at?: string
}
```

再补两张辅助表：

- `memory_write_events`：记录哪次对话、哪个 Agent、为什么写入记忆。
- `conversation_summaries`：保存 thread 压缩摘要，替代无限增长的 `messages`。

如果使用现有 PostgreSQL，可以先落普通表；如果后续要语义检索，再给 `memory_items.content` 增加 embedding，接入 PGVector。

## **四、上下文组装机制**
每次调用模型前，不再直接塞完整 state，而是构造：

```ts
type ContextBundle = {
  runtime_context: object
  current_task: object
  recent_messages: Message[]
  thread_summary?: string
  relevant_memories: MemoryItem[]
  relevant_knowledge_chunks: RagChunk[]
  relevant_artifacts: Artifact[]
  constraints: string[]
  token_budget: object
}
```

建议引入 `ContextAssembler`：

1. 读取当前 `thread_id` 的 checkpoint。
2. 判断任务类型：普通聊天、教学设计、附件分析、产物生成、产物修订。
3. 按任务选择记忆范围。
4. 检索相关长期记忆。
5. 检索 RAG 知识片段。
6. 注入当前 artifact 摘要。
7. 做 token budget 控制。
8. 生成最终 prompt messages。

关键原则：**记忆默认不全量注入，只注入摘要和高相关片段。**

## **五、会话记忆增强**
当前 `messages` 会持续增长，建议加三件事：

1. **消息裁剪**  
   保留最近 N 轮完整消息，例如最近 8-12 轮。

2. **会话摘要**  
   超过阈值后，把早期消息压缩成结构化摘要：

```ts
type ThreadSummary = {
  confirmed_metadata: object
  user_requirements: string[]
  key_decisions: string[]
  open_questions: string[]
  generated_artifacts: string[]
  revision_intents: string[]
}
```

3. **摘要可更新**  
   每轮结束后，如果用户确认了教学要素、设计方案、产物修改目标，就更新 summary，而不是等上下文爆掉才处理。

## **六、长期记忆写入机制**
不要让模型随意“记住一切”。建议采用候选记忆机制：

1. 对话结束后生成 `MemoryCandidate`
2. 按规则筛选
3. 高风险记忆需要用户确认
4. 写入 `memory_items`
5. 后台定期合并、去重、过期

候选记忆结构：

```ts
type MemoryCandidate = {
  content: string
  memory_type: "semantic" | "episodic" | "procedural"
  scope_type: "user" | "plan" | "org"
  reason: string
  confidence: number
  requires_user_approval: boolean
}
```

适合自动记住：

- 教师明确表达的偏好
- 多次重复出现的修改要求
- 已确认的教学计划元数据
- 产物修订中的稳定风格偏好

不应自动记住：

- 学生个人隐私
- 临时情绪表达
- 一次性要求
- 附件中的敏感信息
- 未经确认的模型猜测

## **七、记忆检索策略**
建议做混合检索：

- 精确过滤：`scope_type + scope_id + category`
- 时间权重：近期修改过的记忆优先
- 重要性权重：用户确认过的记忆优先
- 向量相似度：匹配当前任务语义
- 任务路由：不同 graph 节点读取不同记忆

例如：

- 意图识别：只读最近消息、artifact 列表、轻量 thread summary
- 元数据抽取：读 thread summary、plan memory、附件摘要
- RAG 检索：读 plan memory、教材版本、知识库索引
- PPT 生成：读 teacher style、school template、artifact style memory
- 差量修改：读 artifact memory、最近 revision history

## **八、前端产品能力**
需要增加一个“记忆管理”入口，增强信任感：

- 查看 AI 记住了什么
- 手动新增偏好
- 编辑/删除某条记忆
- 关闭某类记忆
- 对“是否记住这个偏好？”进行确认
- 显示当前回答使用了哪些记忆

例如对教师展示：

> 我已记住：你偏好“课堂导入先给生活情境，再提出探究问题”。  
> 作用范围：仅你本人。  
> 可随时删除。

## **九、安全与合规**
教育场景必须特别小心：

- 默认不记学生姓名、联系方式、成绩、家庭情况等个人信息
- 组织级记忆必须由管理员或教研组长确认
- 用户级记忆只能本人和授权管理员访问
- 支持记忆删除、导出、禁用
- 附件解析内容不能自动变成长记忆，除非用户确认
- prompt injection 防护：附件内容只能作为资料，不能写入 procedural memory

## **十、实施路线图**
第一阶段：会话记忆治理，1-2 周

- 增加 `conversation_summaries`
- 为 `messages` 增加摘要压缩策略
- 修复删除 session 时未删除历史状态的问题
- 为 `get_message_histry` 增加摘要/审批/产物状态兼容

第二阶段：长期记忆 MVP，2-4 周

- 新增 `memory_items`
- 实现 `MemoryService`
- 支持用户偏好、教学计划记忆、artifact 修订记忆
- 在教学设计和产物生成 prompt 中注入相关记忆

第三阶段：记忆检索与写入闭环，4-6 周

- 增加 `MemoryCandidateExtractor`
- 增加后台合并、去重、过期任务
- 接入 embedding 检索
- 前端增加“记忆中心”

第四阶段：企业级记忆 Harness，6-10 周

- 记忆权限、审计、删除、导出
- 组织级教学规范记忆
- 记忆命中率、写入准确率、误记率评测
- 与 artifact trace、LLM trace 打通

**最终建议**
SmartClass 的记忆系统不要做成“聊天历史增强”，而要做成：

> 教师偏好记忆 + 教学计划记忆 + 会话工作记忆 + 产物演化记忆 + 学校规范记忆

这样它才会从“每次重新理解老师”升级为“越来越懂这位老师、这个学科、这所学校的备课伙伴”。
