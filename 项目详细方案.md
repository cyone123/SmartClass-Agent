# 项目详细方案

## 一、项目概述

### 1.1 项目名称

智课伴侣 多模态 AI 互动式教学智能体

### 1.2 项目定位

本项目面向教师备课与教学设计场景，致力于构建一个能够理解教师教学思路、融合多模态资料、自动生成多种教学产物，并支持持续修改优化的教学智能体系统。它不是一个单点式问答工具，而是一套以“教学设计驱动”为核心的课件共创与教学准备工作台。

### 1.3 项目目标

项目围绕以下目标展开：

- 将教师从重复性的课件制作和资料整理工作中解放出来
- 通过多轮对话与主动追问，准确理解教师真实教学需求
- 打通“需求输入、知识融合、教学设计、产物生成、迭代优化”的完整链路
- 构建可扩展、可复用、可持续演进的教学智能体工程架构

## 二、背景与问题分析

### 2.1 行业背景

随着教育数字化和生成式 AI 技术的发展，教师对智能化备课工具的需求快速提升。但从实际教学场景看，现有工具普遍存在“功能碎片化”“理解浅层化”“产物不可持续迭代”等问题，难以满足教师对高质量备课的真实需求。

### 2.2 现有问题

当前教学工具主要存在以下痛点：

- 工具割裂：教学设计、PPT 制作、教案撰写、互动内容设计需要在多个系统中分别完成
- 需求表达复杂：教师的教学思路往往需要多轮澄清，而不是一句简单指令
- 资料利用效率低：PDF、Word、视频等参考资料难以直接参与生成链路
- 结果修改困难：大多数工具只能“重新生成”，无法基于已有产物持续优化
- 工程闭环不足：很多产品只能生成文本建议，无法形成可交付文件与在线预览闭环

### 2.3 需求判断

本项目认为，教师真正需要的不是“再一个 AI 工具”，而是一个能够围绕教学思路组织工作的智能系统。系统应当同时具备：

- 多模态输入理解能力
- 结构化教学要素抽取能力
- 本地知识库检索增强能力
- 多种教学产物的真实生成能力
- 产物修改与版本管理能力
- 可视化、可协作、可回溯的前端交互能力

## 三、需求分析

### 3.1 用户对象

项目主要服务以下用户群体：

- 中小学及职业教育教师
- 教研组与课程设计人员
- 学校信息化管理与资源建设部门
- 教育平台及云课堂运营方

### 3.2 核心使用场景

系统面向的典型场景包括：

- 教师进行单课时备课与教学设计
- 根据教学资料快速生成 PPT 与教案
- 基于已有教案、课件、视频资料进行课堂内容重构
- 生成互动教学演示或小游戏
- 针对已生成课件或教案进行定向修订

### 3.3 功能需求拆解

结合命题要求与项目实际，系统需求可拆解为五类：

#### 1. 多模态输入需求

- 支持文字输入与语音输入
- 支持 PDF、Word、视频等参考资料上传
- 建立教师需求、附件内容与生成结果之间的关联

#### 2. 教学意图理解需求

- 支持多轮对话
- 主动追问不完整信息
- 结构化提取学科、年级、主题、课时、重点难点、教学目标等要素

#### 3. 知识融合需求

- 构建本地知识库
- 对知识库文件进行切分、向量化、检索
- 将教师意图、参考资料和知识库上下文统一融合

#### 4. 产物生成需求

- 生成 PPT 课件
- 生成 Word 教案
- 生成互动内容，至少支持一种
- 支持标准格式产物输出

#### 5. 迭代优化需求

- 支持预览
- 支持提出修改意见
- 支持针对已有产物进行差量修改
- 支持新版本结果管理与继续使用

### 3.4 非功能需求

系统还需满足以下工程性要求：

- 工作流可中断、可恢复
- 数据链路清晰，知识库文件与会话附件严格分离
- 前后端协议统一，支持流式事件推送
- 支持多平台友好运行
- 结果具备可追踪性、可预览性和可回写能力

## 四、总体解决思路

本项目采用“工作流驱动 + 多模态融合 + 产物闭环”的总体思路：

1. 通过多模态输入采集教师需求和参考资料  
2. 通过 LangGraph 对教学意图进行结构化理解与流程编排  
3. 通过本地知识库 RAG 和附件分析结果补强上下文  
4. 生成教学设计方案，并通过审批节点与教师协作确认  
5. 基于确认后的方案生成 PPT、DOCX 和互动内容  
6. 对生成产物提供在线预览、回写和差量修改能力  

这一思路解决了传统 AI 教学工具“只负责生成、不负责流程”的问题，使系统真正贴近教师使用方式。

## 五、总体架构设计

系统采用前后端分离架构，可分为四层：

### 5.1 交互层

由前端承接用户交互，主要包括：

- 教学计划与会话管理
- 对话输入与语音录音
- 附件上传
- 进度与审批展示
- 产物查看与预览

### 5.2 工作流编排层

由 LangGraph 负责教学流程控制，包括：

- 意图识别
- 教学元数据提取
- 主动追问
- 教师确认
- RAG 检索
- 教学设计生成
- 产物生成与修订

### 5.3 能力服务层

由多个能力模块提供支撑：

- 大模型调用模块
- RAG 运行时
- 附件分析 Agent
- Skill Registry
- Workspace 执行环境
- 语音转写运行时
- 视频理解运行时
- Artifact 产物生成与修订服务

### 5.4 数据存储层

由数据库、文件存储和向量表组成：

- PostgreSQL 存储计划、会话、文件、artifact 元数据
- LangGraph checkpoint 存储会话状态与恢复点
- 本地存储目录保存知识库文件、附件、artifact 产物和工作区文件
- 向量库保存知识切片与检索数据

## 六、核心技术路线

### 6.1 技术栈选型

项目采用如下核心技术栈：

- 后端：FastAPI、LangChain、LangGraph、SQLAlchemy、PostgreSQL
- 检索：PGVector 风格向量检索
- 模型：OpenAI 兼容大模型接口、Embedding、语音转写模型、视觉模型
- 前端：Vue 3、Vite、Pinia、Element Plus
- 文档预览：OnlyOffice
- 多模态处理：FFmpeg、PDF/DOCX 解析脚本

### 6.2 技术选型原因

选择上述技术的原因在于：

- FastAPI 和 SQLAlchemy 能够快速构建稳定的异步服务
- LangGraph 适合表达可中断、可恢复、可追踪的多步骤工作流
- LangChain 便于组织模型、工具和 Agent 执行逻辑
- PostgreSQL 既可承载业务数据，又适合与向量检索结合
- Vue 3 + SSE 适合构建高响应、强交互的工作台界面
- OnlyOffice 可以支撑真实 Office 文档预览和编辑回写

## 七、详细功能方案与实现设计

### 7.1 教学计划与会话组织方案

为了让教师的备课过程具备清晰边界，系统首先构建了“教学计划 - 会话 - 文件/产物”的基础组织结构。

其中：

- `Plan` 对应一个教学主题或课程单元，承载知识库资料和多个会话
- `Session` 对应一次具体的备课或修改过程
- `thread_id` 作为 LangGraph 的上下文主键，贯穿多轮对话、审批中断、历史状态恢复和产物版本管理

在功能上，系统支持：

- 教学计划创建、修改、删除
- 会话创建、切换、重命名、删除
- 历史消息加载
- 当前线程下 artifact 自动关联显示

这样的设计确保了“知识资料”“对话状态”“产物结果”三者在数据归属上保持一致，也为后续 RAG 检索、附件分析和 artifact 修订提供了稳定锚点。

### 7.2 多模态交互式需求输入方案

系统在输入层面坚持“低门槛表达 + 真实场景适配”的设计原则，为教师提供多种需求表达方式。

#### 1. 文字输入

教师可以直接输入教学目标、课程主题、教学重点、课时安排、课堂活动设想等自然语言描述。系统并不要求教师使用固定模板，而是通过后端结构化理解能力自动抽取有效信息。

#### 2. 语音输入

前端支持浏览器内录音。录音结束后，系统将音频以附件形式提交到后端转写接口，完成转写后自动将文本回填到输入框。该能力适用于教师快速口述备课思路，降低输入门槛。

#### 3. 资料上传

系统支持两类资料接入：

- 知识库文件：用于长期保存和后续检索增强
- 会话附件：用于当前对话轮次的即时理解与生成

目前已支持的文件类型包括：

- 知识库文件：PDF、DOCX、TXT、MD、CSV、JSON
- 会话附件：PDF、DOCX、视频

这种双链路设计的意义在于：

- 知识库文件服务于“课程级、长期”知识沉淀
- 会话附件服务于“本轮任务级、即时”上下文补充
- 避免把一次性参考资料错误写入长期知识库
- 避免长期知识库和当前对话附件在语义上互相污染

### 7.3 教学意图理解与结构化抽取方案

本项目并不把教师输入直接送进生成模型，而是先进行结构化理解，以提高可控性与结果质量。

#### 1. 意图识别

系统首先通过结构化输出判断本轮请求属于以下哪一类：

- 普通聊天
- 教学设计请求
- 产物修改请求

这一层判断可以防止系统把“继续聊天”“修改现有产物”和“新建教学设计”混为一谈。

#### 2. 教学元数据抽取

对于教学设计请求，系统会进一步抽取：

- 学科
- 年级
- 主题
- 课时/时长
- 核心知识点
- 教学重点
- 教学难点
- 教学目标

系统采用结构化 schema 输出，而不是自由文本解析，其优势在于：

- 输出字段稳定
- 便于前后端统一呈现
- 便于后续 RAG 查询构造和产物生成输入
- 可作为审批卡片内容直接展示给教师确认

#### 3. 主动追问与缺失补全

如果抽取结果不完整，系统会进入主动追问节点，针对缺失信息发起针对性提问，例如：

- 课程具体年级
- 课时长度
- 重点难点
- 是否需要生成 PPT、教案或互动内容

只有当教学要素达到可执行程度后，系统才会进入下一步。这一机制有效降低了“需求不清导致的错误生成”。

### 7.4 本地知识库建设与 RAG 检索方案

本项目将本地知识库作为教师备课的重要辅助基础，而不是简单依赖通用大模型知识。

#### 1. 入库流程

知识库文件链路如下：

1. 教师上传文件  
2. 文件写入本地存储目录  
3. 后端将文件加入异步入库队列  
4. `FileIngestionRuntime` 后台消费队列  
5. 根据文件类型完成解析、切分和向量化  
6. 写入向量库，并更新文件状态  

#### 2. 文件处理策略

当前系统支持的处理方式包括：

- PDF：按页抽取文本并切分
- DOCX：解析 `word/document.xml` 获取内容
- TXT/MD/CSV/JSON：进行编码兜底读取后切分

#### 3. 检索增强策略

在教学设计阶段，系统会将：

- 教师当前输入
- 已抽取的教学元数据
- 会话内历史关键信息

拼接为检索查询，并按 `plan_id` 过滤向量结果。这保证了不同教学计划之间的知识边界不会混淆，也让最终生成结果更贴近本地课程资料和教学语境。

### 7.5 会话附件解析与多模态资料融合方案

除知识库外，教师还可以在会话中上传即时参考资料，系统会根据附件类型进入不同分析链路。

#### 1. 文档类附件解析

对于 PDF、DOCX 等文档类附件，系统会启动附件分析 Agent，并根据文件类型自动加载对应 skill，例如：

- `pdf`：提取 PDF 文本与教学相关内容
- `docx`：抽取 Word 文档中的结构与正文信息

Agent 并不是直接凭空分析，而是通过：

- 读取 skill 说明
- 调用 skill 资源
- 运行 skill 脚本

完成面向教学场景的内容摘要。

#### 2. 视频附件解析

对于视频附件，系统会进入视频理解链路，主要包括：

- 使用 FFmpeg 抽取音频
- 调用语音转写模型获取音频文本
- 抽取关键帧
- 调用视觉模型生成关键画面摘要

最终得到的结果既包含“视频中讲了什么”，也包含“视频中展示了什么”，适合教师上传实验演示视频、讲课片段或课堂案例视频。

#### 3. 上下文融合方式

无论来源于文档附件还是视频附件，系统都会将解析结果统一整理成“可注入上下文”，并作为图工作流输入的一部分参与后续教学设计与生成。这样可以保证多模态资料真正进入生成链路，而不是停留在上传展示层。

### 7.6 教学设计主工作流与人机协同方案

本项目的核心并不是单次生成，而是通过图工作流把人机协同过程显式组织起来。

教学设计主流程如下：

1. 接收教师输入、附件信息和会话上下文  
2. 执行意图识别  
3. 进入教学元数据抽取  
4. 信息不足时主动追问并中断等待用户输入  
5. 教师确认教学要素  
6. 系统执行本地知识库 RAG 检索  
7. 基于教学要素和检索结果生成教学设计方案  
8. 教师确认教学设计方案  
9. 进入多产物生成阶段  

这一设计的优势在于：

- 教师始终参与关键决策节点
- 工作流状态可中断并恢复
- 系统可根据不同状态进入不同分支
- 前端能够真实展示每一步处理进度

因此，系统呈现出的不是“AI 自动替教师完成全部工作”，而是“教师与 AI 协同完成教学设计”。

### 7.7 多模态课件生成引擎方案

在完成教学设计方案确认后，系统进入产物生成阶段。当前已实现三类教学产物生成能力。

#### 1. PPT 课件生成

系统通过 `ppt-generator` skill 与工作区脚本协同生成真实 `.pptx` 文件。生成内容一般包含：

- 封面页
- 目录页
- 核心知识讲解页
- 活动设计页
- 总结页

此类产物重点满足教师对可编辑课件的需求。

#### 2. DOCX 教案生成

系统通过 `docx` skill 生成真实 `.docx` 文件，内容可覆盖：

- 教学目标
- 教学重点与难点
- 教学过程设计
- 教学方法
- 课堂活动设计
- 课后作业

#### 3. HTML 互动内容生成

系统通过 `html-interactive` skill 生成单文件 HTML 页面，用于实现：

- 互动教学演示
- 小游戏
- 概念实验页
- 参数调节式演示内容

#### 4. 选择性生成机制

在前端审批卡片中，教师可以选择需要生成的产物类型，例如：

- 只生成 PPT
- 只生成 DOCX 教案
- 同时生成 PPT、教案和互动内容

这使系统不再是“固定输出”，而是具备更符合真实需求的灵活交付方式。

### 7.8 Artifact 产物管理与差量修改方案

项目将所有生成结果统一抽象为 artifact，并围绕 artifact 建立了一整套管理机制。

#### 1. Artifact 统一模型

每个 artifact 均具备以下关键属性：

- 产物类型
- 标题
- 当前状态
- 所属 `plan_id`
- 所属 `thread_id`
- 文件路径
- MIME 类型
- 版本链信息

#### 2. 产物状态管理

系统对产物设置了清晰状态：

- pending
- running
- ready
- failed

前端会根据后端实时推送的状态变化更新界面展示。

#### 3. 差量修改流程

针对教师对已有产物提出的修改意见，系统会：

1. 判断当前请求是否为 artifact revision  
2. 识别修订对象是 PPT、DOCX 还是 HTML 内容  
3. 如目标不明确，则通过审批节点要求用户澄清  
4. 将源产物复制进工作区  
5. 自动生成源产物摘要  
6. 对 Office 文件进行解包，供 Agent 进行结构参考  
7. 调用对应 skill 和工作区工具执行定向修改  
8. 生成带版本关系的新 artifact  

#### 4. 版本链管理

系统会在数据库中记录：

- `parent_artifact_id`
- `root_artifact_id`
- `revision_number`
- `is_current`

从而支持：

- 追踪当前产物来源
- 标识最新版本
- 保留历史版本链

这一能力使系统从“生成工具”升级为“教学内容持续优化平台”。

### 7.9 前端工作台、SSE 协议与过程可视化方案

前端采用三栏式教学工作台设计：

- 左侧：教学计划与会话组织
- 中间：教学对话与流程交互
- 右侧：知识库文件与当前会话产物

#### 1. SSE 流式通信

系统通过 `/api/chat/stream` 建立 SSE 流式返回，当前前后端已支持以下事件类型：

- `metadata`
- `progress`
- `token`
- `artifact`
- `artifact_trace`
- `approval`
- `suggestions`
- `error`
- `done`

#### 2. 前端承接能力

基于这些事件，前端实现了：

- AI Markdown 实时渲染
- 进度步骤卡片
- 审批卡片与确认/取消操作
- 产物状态实时更新
- 产物生成轨迹可视化
- 后续追问建议展示

#### 3. 工程价值

这一前后端协议设计的价值在于，系统呈现出来的不是“问一句回一句”的聊天页面，而是“可观察的智能工作流界面”。用户既能看到结果，也能看到过程、状态和协同节点。

### 7.10 文档预览、在线编辑与回写方案

为了保证产物真正可用，系统接入 OnlyOffice 作为文档预览与编辑组件。

#### 1. 预览能力

系统当前支持：

- 知识库文件预览
- Artifact 文件预览
- HTML 互动内容 iframe 预览

对于 PPT、DOCX、PDF 等 Office / 文档类文件，系统会返回 OnlyOffice 所需配置并在前端初始化编辑器。

#### 2. 回写能力

OnlyOffice 编辑完成后，后端会接收回调并执行不同处理：

- 若为知识库文件，则重置状态并重新入库
- 若为 artifact，则更新文件内容并保持其可用状态

#### 3. 闭环意义

这一能力意味着系统不仅能自动生成第一版结果，还支持教师在平台内继续人工编辑，随后把编辑后的内容重新纳入系统链路，形成“AI 生成 - 人工编辑 - 系统回收”的闭环。

## 八、关键创新点与特色

### 8.1 从“对话工具”升级为“教学工作流智能体”

本项目的首要创新，不在于简单接入大模型做教学问答，而在于将教师备课过程拆解为一条可执行、可中断、可恢复、可协作的工作流。系统围绕教师真实使用路径，将“需求输入—教学要素提取—主动追问—知识融合—教学设计—产物生成—产物修订”组织成显式图流程，使 AI 能够真正参与教学设计过程，而不是停留在一次性文本回复层面。

与常见的单轮 prompt 式产品相比，本项目的优势在于：

- 能够逐步澄清教师意图，而不是模糊生成
- 能够在关键节点等待教师确认，增强可控性
- 能够在后续继续进入修订流程，而不是推倒重来
- 能够把多模态资料、知识库检索和产物生成放在统一工作流中协调处理

这使系统更接近真实教学生产工具，而非通用型聊天应用。

### 8.2 借鉴 Anthropic 风格 Skills 思路，构建可扩展的渐进式披露能力

项目在 Agent 能力设计上，借鉴了 Anthropic 风格的 Skills 组织思想，并结合当前 LangChain/LangGraph 运行时，自主实现了基于 `SKILL.md` 的技能注册与按需加载机制。系统并不是在初始化时一次性把全部工具说明和资源注入模型，而是采用“先发现、再加载、再逐步读取资源或执行脚本”的渐进式披露模式。

这一机制具备几个明显优势：

- 降低上下文污染，避免将大量无关说明一次性塞入 prompt
- 提升可扩展性，新增 skill 时只需补充技能目录和资源描述
- 让 Agent 根据任务类型动态装配能力，增强执行针对性
- 便于沉淀领域技能资产，例如 PDF 解析、DOCX 编辑、PPT 生成、HTML 互动内容生成等

在当前项目中，系统已经实现：

- `SkillRegistry`：解析技能目录、前置信息、兼容性说明和允许工具
- `load_skill`：按需加载技能说明
- `read_skill_resource`：逐步读取参考文档、模板或资源文件
- `run_skill_script`：调用技能内置脚本完成专业处理

这种技能机制使系统具备从“单体 Agent”向“领域能力库型 Agent 平台”演进的基础。

### 8.3 自主实现安全可控的 Agent 工作区

为了让 Agent 具备真实文件生成与修订能力，而不是只输出一段文本，本项目自主设计并实现了受控的 Agent Workspace。每个线程和运行任务都会映射到独立的工作区目录，包括工作目录、运行目录和输出目录，供 Agent 在安全边界内读写文件、执行脚本和产出结果。

该工作区并非开放式 shell，而是通过自定义工具进行精细控制，包括：

- 列出工作区文件
- 读取工作区文件
- 写入工作区文件
- 精准替换文件内容
- 运行 Python 或 Node 代码

同时，系统加入了多重安全约束：

- 禁止路径穿越，防止访问工作区外部文件
- 禁止在工作区内执行依赖安装命令
- 限制输出长度与执行行为
- 通过技能权限控制哪些 skill 才能调用工作区工具
- 对代码准备与执行过程进行进度上报

这一设计既保留了 Agent 的“真实执行能力”，又避免了不可控的系统级风险，是项目在工程安全与能力开放之间做出的重要平衡。

### 8.4 多模态输入统一抽象为“可注入上下文”

项目并没有把 PDF、DOCX、语音、视频等输入分别硬编码成互不兼容的特例，而是建立了统一的多模态输入抽象：

`原始文件 -> 解析结果 -> 可注入上下文`

在这一设计下：

- 知识库文件进入长期检索链路
- 会话附件进入当前轮对话的即时分析链路
- 语音输入转为文本
- 视频输入转为音频转写与关键帧摘要

最终所有结果都会被转成工作流和 Prompt 可直接消费的上下文对象。这样的统一抽象降低了系统复杂度，也为后续扩展图片、PPT、更多视频理解能力预留了清晰接口。

### 8.5 结构化输出 + 审批中断，增强结果质量与可控性

在教学场景中，需求模糊和结果失真是大模型应用的核心难点。本项目通过“结构化输出 + 审批中断”的组合方案，显著提升了生成质量和交互可靠性。

系统并不直接让模型自由输出教学要素，而是先通过结构化 schema 抽取：

- 学科
- 年级
- 主题
- 课时
- 核心知识点
- 重点、难点
- 教学目标

当信息不完整时，系统会自动进入追问阶段；当信息完整时，会通过审批卡片请教师确认，再继续进入 RAG 检索和教学设计生成。该模式带来的价值在于：

- 减少误解需求导致的无效生成
- 让教师在关键节点保持控制权
- 让工作流具备清晰状态和恢复能力
- 为后续生成与修订打下稳定数据基础

这体现了项目对“高质量生成”问题的工程化处理，而不是仅依赖模型一次性猜测。

### 8.6 真实可交付产物闭环，而非停留在文本建议

本项目的重要特色在于，系统最终交付的不是一段建议说明，而是真实可使用的教学产物。当前已实现的产物包括：

- `.pptx` 课件
- `.docx` 教案
- `.html` 互动内容

更进一步，项目还构建了完整的产物生命周期管理：

- 生成中的 artifact 状态记录
- 生成完成后的统一展示
- 在线预览与打开
- 文档编辑回写
- 新旧版本关系记录

这使项目具备明显的落地属性，也让比赛评审能够看到“从模型输出到业务交付”的完整工程链路。

### 8.7 产物差量修改与版本链管理

相较于常见 AI 工具“修改即重来”的模式，本项目已经实现较具特色的产物差量修改能力。系统会根据教师反馈判断修改目标，并将已有产物作为源对象导入工作区，结合用户新的修改要求生成新版本结果。

该能力包含多个工程创新点：

- 自动识别当前请求是否为“产物修改”
- 自动推断目标产物类型或要求用户澄清
- 将源文件复制到工作区，供 Agent 参考或解包处理
- 为源产物生成结构摘要，辅助定向修改
- 对 Office 文件保留 XML 解包参考信息
- 在数据库中维护父版本、根版本、当前版本和修订号

这一设计让系统从“生成器”升级为“教学内容协同编辑系统”，对教师场景的适配度更高，也具有较强的展示亮点。

### 8.8 基于 Skills 与专业脚本的产物生成引擎

项目没有直接让大模型裸生成二进制文档，而是构建了“Skill + Script + Workspace”的产物生成方式：

- PPT 生成由 `ppt-generator` skill 负责
- Word 教案生成由 `docx` skill 负责
- HTML 互动内容生成由 `html-interactive` skill 负责
- PDF/DOCX 附件分析由相应 skill 和脚本协同处理

这种方式具备明显优势：

- 将专业文件处理逻辑从 prompt 中剥离出来
- 增强生成稳定性和可调试性
- 降低模型直接处理复杂二进制格式的风险
- 便于后续继续沉淀模板、脚本和行业知识

从工程角度看，这代表项目已从“模型中心”转向“模型 + 工具 + 领域技能”协同驱动。

### 8.9 过程透明、结果可追踪、前端可视化程度高

项目不仅重视结果，也重视过程的可观察性。前端围绕 SSE 事件流实现了多层次过程可视化，包括：

- Token 流式输出
- 流程进度卡片
- 审批卡片
- Artifact 状态变化
- Artifact Trace 轨迹卡片
- 建议追问

特别是 `artifact_trace` 机制，能够展示产物生成过程中的状态变化、AI 输出、工具调用和工具结果。这对比赛展示、项目调试、后续优化都具有很高价值。

它带来的收益包括：

- 让系统过程从黑盒变为白盒
- 让教师更容易理解系统当前正在做什么
- 让开发者更容易排查生成链路中的问题
- 提升产品的可信度与交互体验

### 8.10 AI 生成、人工编辑、系统回写三者闭环打通

项目引入 OnlyOffice，对 PPT、DOCX 和 PDF 类文件提供在线预览与编辑能力。更重要的是，编辑后的结果不是停留在浏览器端，而是会通过回调写回系统，并继续进入后续链路：

- 知识库文件可重新入库
- Artifact 可更新为新的内容状态

这使得本项目形成了“AI 先生成、教师再编辑、系统继续接管”的完整闭环。相比纯生成式系统，这一能力更符合真实办公和教学场景，也更有产品落地价值。

- 审批卡片、进度卡片、产物轨迹卡片展示
- 文件上传、预览与会话状态承接
- 用户演示体验与视觉组织

该角色决定系统在比赛展示中的完成度和“产品感”。

#### 5. 测试与交付方向

负责：

- 主链路联调测试
- 边界情况与异常链路验证
- 演示用样例与脚本准备
- 比赛材料、PPT、视频、答辩文案整理

该角色重点保障“能稳定演示、能清晰表达、能顺利交付”。

### 11.3 实施阶段建议

项目实施可分为四个阶段：

#### 第一阶段：需求分析与总体设计

目标：

- 完成场景梳理与功能拆解
- 明确系统对象模型与数据链路
- 选定技术栈与总体架构

输出：

#### 第二阶段：核心链路开发

#### 第三阶段：功能增强与亮点建设

这一组织方式能够确保项目不仅“技术上能做”，也“协作上做得成、展示上讲得清”。

## 十二、风险分析与优化方向

### 12.1 主要风险

#### 1. 多模态资料解析稳定性风险

随着支持的资料类型增加，不同格式文件在编码、结构、内容质量上的差异会导致解析效果波动。例如：

- PDF 文本抽取质量受原始排版影响较大
- DOCX 中可能存在复杂样式、批注、图文混排
- 视频中音频噪声和画面复杂度会影响摘要质量

#### 2. 教学意图理解偏差风险

教师需求天然具有模糊性和多轮演化特征，如果系统在早期理解阶段发生偏差，后续生成结果可能整体偏离预期。

#### 3. 生成结果一致性风险

在复杂教学风格、版式要求、格式约束下，生成的 PPT、教案和互动内容可能存在：

- 结构完整但风格不够贴合
- 内容合理但表达不够精炼
- 修改后新旧版本一致性不足

#### 4. Agent 工作流时延与成本风险

项目采用多步骤工作流、RAG、多模态解析和真实文件生成，这会在复杂场景下带来一定时延和资源开销。如果不做约束，可能影响用户体验与部署成本。

#### 5. 外部依赖与部署兼容风险

项目依赖模型接口、FFmpeg、OnlyOffice、数据库和宿主环境。不同答辩环境、服务器环境或学校环境下，可能存在：

- 模型接口不可达
- 文档预览环境差异
- Office 回写兼容性问题
- Node / Python 宿主依赖差异

### 12.2 对应策略

#### 1. 统一抽象与协议治理

继续强化以下统一抽象：

- 多模态输入抽象
- artifact 对象抽象
- progress 步骤抽象
- SSE 事件协议

通过协议统一降低模块耦合，减少功能增强时的返工成本。

#### 2. 用结构化输出与审批机制控制质量

对于意图识别和教学元数据提取，继续坚持结构化 schema 输出；对于关键节点，继续采用教师确认与补充输入机制。这样可以将错误尽量拦截在生成之前，而不是在生成之后被动返工。

#### 3. 用 Skill 与 Workspace 机制约束 Agent 行为

继续通过 Skills 权限、工作区边界和脚本约束来限制 Agent 的执行范围，避免不可控操作，提高生成过程稳定性、可解释性和可调试性。

#### 4. 分层优化时延与成本

针对工作流较长带来的性能问题，可采用：

- 轻量模型优先处理结构化理解任务
- 生成前先做明确审批，避免无效生成
- 选择性生成产物，控制资源消耗
- 通过缓存和工作区复用减少重复计算

#### 5. 加强样例测试与演示预案

围绕比赛重点场景，构建稳定的测试和演示样例，包括：

- 标准备课样例
- 文档附件样例
- 视频附件样例
- 产物修改样例

同时准备兜底预案，例如演示环境异常时切换到备用样例或预生成结果，以确保展示稳定。

#### 6. 明确近期优化方向

后续重点优化方向包括：

- 增强图片和 PPT 参考资料解析
- 增强风格继承与版式控制能力
- 丰富互动内容与动画创意生成
- 提升 artifact 差量修改的精细度
- 增强部署适配性与工程稳定性

通过这些策略，项目可以在保持当前亮点的基础上持续提升质量和完整度。

## 十三、总结

“智课伴侣” 项目围绕教师真实备课流程，构建了一套具备多模态输入、教学意图理解、本地知识库增强、教学设计规划、真实产物生成、在线预览与差量修改能力的教学智能体系统。项目不是把大模型简单接入到对话框中，而是围绕“需求理解—知识融合—工作流编排—产物生成—产物迭代”的完整链路进行系统设计和工程实现。

本项目不仅展示了大模型、多模态处理和 RAG 等技术在教育场景中的组合应用，更重要的是展示了将这些能力工程化落地为真实产品原型的能力。它证明了 AI 在教学场景中可以不只是“回答问题”，而是能够真正参与教师的备课设计、课件共创和内容持续优化过程。

未来，项目还可以继续沿着“更多模态理解、更强风格控制、更丰富产物类型、更稳定部署能力”的方向演进，进一步从比赛原型发展为面向学校、教研组和教育平台的教学智能工作平台。
